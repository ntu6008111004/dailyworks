# 🗓️ บันทึกสรุปการปรับปรุงระบบแบบละเอียด (28 พฤษภาคม 2026)

วันนี้ทีมพัฒนาได้ปรับปรุงโครงสร้างระบบจัดเก็บข้อมูลและแก้ไขข้อผิดพลาดหลังการอัปเกรดระบบบันทึกงาน โดยสรุปรายละเอียด ปัญหาที่พบ และแนวทางแก้ไขเพื่อการรายงานความคืบหน้าดังนี้ครับ:

---

## 🚀 1. การย้ายฐานข้อมูลและเชื่อมโยงระบบ (Database Migration)
- **ทำอะไรไปบ้าง:**
    - ทำการย้ายระบบจัดเก็บข้อมูลหลักจาก **Google Sheets** ที่มีข้อจำกัดในการประมวลผล ไปสู่ **Supabase (PostgreSQL Cloud Database)** ซึ่งเป็นระบบฐานข้อมูลแบบคลาวด์ประสิทธิภาพสูง
    - ทำการโอนย้ายชุดข้อมูลเดิมทั้งหมดขององค์กร ประกอบไปด้วย:
        - **ตารางข้อมูลตำแหน่งพนักงาน (Positions):** จำนวน 10 รายการ
        - **ตารางข้อมูลรายชื่อพนักงาน (Users):** จำนวน 15 รายการ (พร้อมสิทธิ์การเข้าใช้งาน)
        - **ตารางข้อมูลบันทึกงานประจำวัน (Tasks):** จำนวน 1,859 รายการ
        - **ตารางข้อมูลคำสั่งบรีฟงาน (Briefings):** จำนวน 269 รายการ
        - **ตารางประวัติการตอบรับคำสั่งบรีฟ (BriefingResponses):** จำนวน 224 รายการ
- **ทำไมต้องย้าย (สาเหตุ):**
    - เนื่องจากระบบเดิมดึงข้อมูลตรงจาก Google Sheets ผ่าน Google Apps Script (GAS) ซึ่งเมื่อมีพนักงานใช้งานพร้อมกันจำนวนมาก หรือมีจำนวนแถวข้อมูลเกิน 1,000 รายการ จะส่งผลให้หน้าเว็บโหลดข้อมูลช้ามาก (ใช้เวลาเกิน 10 วินาทีขึ้นไป) มีอาการหน้าจอค้าง และบ่อยครั้งระบบ Google จะปฏิเสธการเชื่อมต่อชั่วคราว (Rate Limit) ทำให้ไม่สามารถบันทึกหรือค้นหางานได้
- **ผลลัพธ์หลังย้าย:**
    - การบันทึกงาน, การแก้ไขสถานะงาน และการค้นหาทำงานได้รวดเร็วทันที (Real-time) ลดเวลาหน่วงการดึงข้อมูลจาก 10 กว่าวินาที เหลือไม่ถึง 1 วินาที

---

## 📸 2. การเพิ่มและยกระดับระบบแนบรูปภาพ (High-Resolution Image System)
- **ทำอะไรไปบ้าง:**
    - ปรับปรุงระบบให้รองรับการเก็บภาพแนบในผลงาน (สูงสุด 4 รูปต่อ 1 บันทึกงาน) ที่มีความคมชัดสูงขึ้น (High Resolution)
    - การเก็บไฟล์ภาพเดิมบน Google Sheets ทำให้ไฟล์เอกสารบวมและทำให้การส่งข้อมูลโหลดช้ามาก เราจึงเปลี่ยนมาเก็บภาพในรูปแบบ Base64 ลงในฐานข้อมูลคลาวด์ Supabase โดยตรง
    - **การเพิ่มความเร็วเฉพาะจุด (Optimization):** เพื่อป้องกันไม่ให้หน้าแสดงรายการงานโหลดช้าจากการต้องดาวน์โหลดรูปภาพขนาดใหญ่พร้อมกันนับพันรูป เราได้สร้าง **`TasksSummary` View** ในฐานข้อมูล เพื่อดึงเฉพาะข้อความรายละเอียดของงานมาแสดงผลก่อน (โดยมีสถานะบอกว่างานนี้มีรูปแนบหรือไม่) และจะไปดึงไฟล์ภาพความละเอียดสูงมาแสดงเฉพาะตอนที่พนักงานกดเปิดคลิกดูรายละเอียดของงานชิ้นนั้นๆ (Lazy Loading)
- **ผลลัพธ์:**
    - หน้าจอรายการงานหลักยังคงโหลดด้วยความเร็วสูงสุด แม้จะมีรูปภาพแนบในระบบจำนวนมากก็ตาม และพนักงานสามารถดูรูปภาพรายงานผลงานที่คมชัดและละเอียดขึ้นได้

---

## ⚠️ 3. ปัญหาที่พบหลังการย้ายระบบ และวิธีการแก้ไข (Issues & Post-Migration Fixes)

หลังจากการย้ายระบบและปรับโครงสร้าง SQL พบปัญหาหลัก 3 ประการ ซึ่งทีมงานได้ทำการวิเคราะห์และแก้ไขเรียบร้อยแล้ว ดังนี้:

### 🔴 ปัญหาที่ 1: ยอดสรุปจำนวนงานรายเดือนแสดงผลไม่ตรง (งานขาดหายไปจำนวนมาก)
- **สาเหตุ:** 
    - **ข้อจำกัดการดึงข้อมูลของ Supabase (Default Row Limit):** โดยระบบความปลอดภัยของ Supabase จะตั้งค่าจำกัดการดึงข้อมูลไว้ที่ 1,000 แถวต่อครั้ง เพื่อป้องกันไม่ให้ฐานข้อมูลทำงานหนักเกินไป แต่ในระบบของเรามีข้อมูลจริงสะสมอยู่ถึง **1,859 รายการ** ส่งผลให้ระบบดึงข้อมูลมาไม่ครบ (หายไป 859 รายการที่บันทึกไว้ในอดีต) ทำให้ยอดสรุปงานรายเดือนของโปรแกรมเมอร์ในเดือนเมษายน แสดงผลเพียง 29 งาน จากยอดจริง 149 งาน
    - **การกรองวันที่ผิดพลาด (Filter Logic Issue):** โค้ดตัวกรองเดิมใช้เงื่อนไขตรวจสอบงานโดยอิงจาก "วันเริ่มงาน" (StartDate) ร่วมกับ "วันกำหนดส่ง" (DueDate) เท่านั้น หากผู้ใช้ลืมระบุวันกำหนดส่ง หรือกรอกข้อมูลไม่ครบ ตัวระบบจะคัดกรองงานชิ้นนั้นทิ้งทันที
- **แนวทางการแก้ไข:**
    - เขียนชุดคำสั่งเสริมพิเศษ **`fetchAllRows` (Pagination Factory)** เข้าไปควบคุมระบบ API ของหลังบ้าน โดยสั่งให้ระบบวนลูปไปดึงข้อมูลทีละ 1,000 รายการแบบอัตโนมัติจนกว่าจะครบทุกรายการในฐานข้อมูล (ครบ 1,859 รายการ) ทำให้สามารถนำเข้าข้อมูลมาคำนวณสรุปได้ทั้งหมด
    - ปรับปรุงระบบกรองเวลา (Filter Logic) ในหน้าสรุปรายเดือนใหม่ ให้ฉลาดขึ้นโดยครอบคลุม 4 กรณี:
        - กรณีระบุทั้ง วันเริ่ม - วันส่งงาน (คำนวณคาบเกี่ยว)
        - กรณีมีเฉพาะวันเริ่มงาน (แสดงตามเดือนที่เริ่ม)
        - กรณีมีเฉพาะวันส่งงาน (แสดงตามเดือนที่ส่ง)
        - กรณีไม่มีการกรอกวันที่เลย (จะดึงวันที่ทำการกดบันทึก CreatedAt มาเป็นตัวแทนของเดือนนั้นๆ อัตโนมัติเพื่อป้องกันข้อมูลตกหล่น)
- **ผลลัพธ์:** ยอดแสดงสรุปงานรายเดือน และยอดงานทั้งหมดของพนักงานแสดงผลถูกต้อง 100% ครบถ้วนตามข้อมูลจริงแล้ว

### 🔴 ปัญหาที่ 2: ระบบนำเข้าข้อมูลประวัติเดิมจาก Google Sheets หยุดทำงานกลางคัน (Seeding Timeout)
- **สาเหตุ:** 
    - เนื่องจากข้อมูลประวัติการตอบกลับบรีฟงาน (Briefing Responses) ใน Google Sheets มีปริมาณมากและกระจัดกระจาย เมื่อสคริปต์ของ Node.js ยิงคำสั่งดึงข้อมูลขนานกันในปริมาณสูง ทำให้ Server ของ Google เกิดการบล็อกและตัดการเชื่อมต่อ (Aborted/Timeout) ส่งผลให้การโอนถ่ายประวัติงานค้างคาอยู่ที่ Batch 7 จาก 14
- **แนวทางการแก้ไข:**
    - ปรับลดจำนวนการดึงข้อมูลพร้อมๆ กันลงให้อยู่ในอัตราส่วนที่ปลอดภัยต่อการเชื่อมต่อ (ลด BATCH CONCURRENCY เหลือ 5)
    - เสริมระบบ **Retry Mechanism** ร่วมกับ **AbortController** เข้าไปในตัวดึงข้อมูล หากสัญญานเชื่อมต่อจากฝั่ง Google ขาดหายไป ตัวระบบจะทำการเชื่อมต่อและดึงงานชุดเดิมใหม่อัตโนมัติทันทีได้สูงสุด 3 ครั้งโดยไม่เริ่มนับหนึ่งใหม่
- **ผลลัพธ์:** สามารถกู้คืนประวัติงานและการตอบกลับบรีฟงานจากระบบเก่าทั้งหมดเข้ามาสู่ระบบใหม่ได้สมบูรณ์ครบถ้วน 100%

### 🔴 ปัญหาที่ 3: ระบบแจ้งเตือนข้อผิดพลาด Supabase URL หายไปหลังเปิดใช้งานบน Vercel
- **สาเหตุ:**
    - ตัวแปรระบบ (Environment Variables) ที่ใช้สำหรับกำหนดลิงก์เชื่อมต่อฐานข้อมูลในหน้าเว็บ (Frontend) ถูกระบุชื่อเป็น `SUPABASE_URL` และ `SUPABASE_ANON_KEY`
    - ซึ่งระบบสร้างหน้าเว็บของ **Vite** จะยอมรับเฉพาะตัวแปรที่มีการเขียนกำกับว่า **`VITE_`** นำหน้าชื่อเท่านั้นเพื่อความปลอดภัย ส่งผลให้หน้าเว็บบน Vercel ค้นหาลิงก์เข้าสู่ระบบไม่เจอจนระบบปิดกั้นการทำงาน
- **แนวทางการแก้ไข:**
    - แก้ไขชื่อตัวแปรบนระบบ Cloud ของ Vercel ใหม่เป็น **`VITE_SUPABASE_URL`** และ **`VITE_SUPABASE_ANON_KEY`** และทำการสั่งคอมไพล์ระบบและ Deploy ใหม่
- **ผลลัพธ์:** หน้าเว็บเชื่อมต่อกับฐานข้อมูลความเร็วสูงได้ถูกต้อง ปลอดภัย และแสดงผลได้เป็นปกติทุกประการ
