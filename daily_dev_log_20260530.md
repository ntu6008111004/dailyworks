# 🗓️ บันทึกสรุปการปรับปรุงระบบแบบละเอียด (30 พฤษภาคม 2026)

วันนี้ทีมพัฒนาได้แก้ไขจุดบกพร่อง (Bug Fix) สำคัญ 2 จุด ที่ส่งผลกระทบต่อการใช้งานระบบกรองข้อมูลและการเปลี่ยนสถานะงาน พร้อมทั้งปรับปรุงเวอร์ชันระบบจาก 0.0.2 เป็น 0.0.3 โดยสรุปรายละเอียดได้ดังนี้ครับ:

## 🔍 1. แก้ไขระบบกรองข้อมูลตามช่วงวันที่ (Date Range Filter)
- ปัญหาเดิม:
    - เมื่อผู้ใช้กรองงานตามช่วงวันที่ เช่น ใส่วันเริ่มต้นเป็นวันที่ 30 และวันสิ้นสุดเป็นวันที่ 1 ของเดือนถัดไป ระบบจะใช้วันกำหนดส่ง (DueDate) เป็นตัวเปรียบเทียบ ทำให้ระบบตีความผิดว่าเป็นงานของเดือนถัดไปแทนที่จะเป็นเดือนปัจจุบัน
    - สาเหตุ: ตรรกะ Filter ใน `api.js` ใช้ `DueDate` (วันกำหนดส่ง) เป็นเกณฑ์หลักในการเปรียบเทียบช่วงวันที่ แทนที่จะใช้ `StartDate` (วันเริ่มต้นงาน)
- ทำอะไรไปบ้าง:
    - แก้ไขตรรกะการกรองวันที่ในไฟล์ `frontend/src/services/api.js` (บรรทัด 306-314)
    - เปลี่ยนจาก `const compareDate = t.DueDate || t.StartDate` → `const taskStartDate = (t.StartDate || t.DueDate || '').slice(0, 10)`
    - ทำให้ระบบเปรียบเทียบจากวันเริ่มต้นงาน (StartDate) เป็นหลัก และตัดค่าวันที่ให้เป็นรูปแบบ `yyyy-MM-dd` ที่แม่นยำ
- ผลลัพธ์:
    - ผู้ใช้สามารถกรองงานตามช่วงวันที่ได้ถูกต้องตามวันเริ่มต้นจริงของงาน ไม่ถูกตีเป็นเดือนอื่นอีกต่อไป

## 🔄 2. แก้ไขปัญหาสถานะงานดีดกลับหลังเปลี่ยนสถานะ (Status Bounce-Back)
- ปัญหาเดิม:
    - เมื่อผู้ใช้เปลี่ยนสถานะงานหลายรายการ (เช่น จาก "ยังไม่เริ่ม" เป็น "เสร็จสิ้น" จำนวน 6 รายการ) สถานะจะเปลี่ยนสำเร็จในตอนแรก แต่หลังผ่านไปสักครู่ บางรายการจะดีดกลับไปเป็นสถานะเดิม (เช่น 4 จาก 6 รายการกลับเป็น "ยังไม่เริ่ม") ทั้งที่ข้อมูลในฐานข้อมูลเปลี่ยนเรียบร้อยแล้ว (รีเฟรชหน้าจะเห็นค่าถูกต้อง)
    - สาเหตุ: หลังจากทำ Optimistic Update แล้ว ระบบจะรอ 1.5 วินาทีแล้วทำ Silent Re-fetch ข้อมูลจากเซิร์ฟเวอร์ แต่ฟังก์ชัน `updateTaskStatus()` ใน `api.js` ไม่ได้ล้าง Cache เก่าออก (ต่างจากฟังก์ชัน `updateTask()` ที่มีการเรียก `clearCacheFor()`) ทำให้ Silent Re-fetch ดึงข้อมูลจาก Cache เก่าที่ยังเป็นสถานะก่อนแก้ไขมาทับข้อมูลใหม่ที่อัปเดตไปแล้ว
- ทำอะไรไปบ้าง:
    - **ไฟล์ `frontend/src/services/api.js` (บรรทัด 772-785)**:
        - เพิ่ม `this.clearCacheFor('getTasksSummary', 'getTasksPaged', 'getTasks')` หลัง API call สำเร็จ เพื่อล้าง Cache เก่าทิ้งทันที ให้การ Fetch ข้อมูลครั้งถัดไปดึงข้อมูลใหม่จากเซิร์ฟเวอร์จริง
    - **ไฟล์ `frontend/src/views/Tasks.jsx` (บรรทัด 419-442)**:
        - ปรับลำดับการทำงานให้ `await` รอ API call เสร็จก่อน แล้วค่อยแสดง Success Indicator
        - ทุก callback ที่ set state กลับ ให้บังคับใส่ค่า `Status: newStatus` เสมอ เพื่อป้องกันค่าเก่าเข้ามาทับ
        - Silent Re-fetch หลัง 1.5 วินาทีจะดึงข้อมูลจากเซิร์ฟเวอร์จริง (เพราะ Cache ถูกล้างแล้ว)
- ผลลัพธ์:
    - สถานะงานจะไม่ดีดกลับอีกต่อไปหลังเปลี่ยนสถานะ ข้อมูลบน UI จะตรงกับฐานข้อมูลตลอดเวลา

## 🧪 4. การทดสอบและการส่งมอบงาน (Git Operations)
- ทำอะไรไปบ้าง:
    - บันทึกประวัติโค้ดที่แก้ไขผ่านกระบวนการ Git:
        - สั่ง `git commit` การเปลี่ยนแปลงเข้าสู่สาขาพัฒนา (dev) จำนวน 2 commits
        - ทำการ `git push origin dev` ขึ้นสู่ GitHub
        - สลับไปสาขาหลัก (main) ด้วย `git checkout main`
        - ดำเนินการ Merge ด้วย `git merge dev`
        - ทำการ `git push origin main` เพื่อเผยแพร่เวอร์ชัน 0.0.3 ใหม่
        - สลับสาขากลับมายัง dev เพื่อเปิดรับการพัฒนาต่อ
